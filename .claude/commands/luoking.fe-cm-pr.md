5. 注意，串行执行，不允许并行执行


### Commit 规范
严格遵守下面的规范，执行代码的 git commit

- 如果当前在main分支，请帮我新建分支操作
- unstaged，把所有暂存区的内容重新推到工作区
- 从工作区中找到功能独立的变更，作为一条 commit 的内容，使用英文描述
- 尽可能多的分离 commit
- 使用简化版本的 Conventional Commits 规范:
  - 格式如下: `<type>: <description>`
  - 严格根据格式
  - 英文小写
- 严格禁止任何方式添加 Generated by XXX 和 Co-Author 等信息，只能有 <type> 和 <description>
- commit结束你需要push到远程仓库（统一在 webserver 仓库）


### 创建PR规范
严格遵守下面的规范，执行github的 PR 创建

- 先检查是否已经开启了一个pr，如果开启了pr则补充一下内容就好
- 标题使用: <type>:<description> 格式
- 严格的只能简单罗列几条改动描述
- 使用英文描述
- 严格禁止任何方式添加 Generated by XXX 和 Co-Author 等信息
- 在提交PR之前，必须严格重新检查是否违反了这个PR规范
- 如果我给了你issues number，那么请在第一行增加issues标记（独占一行， 比如 我给你 "P-1234", 你应该把 "resolve P-1234" 放在第一行）
